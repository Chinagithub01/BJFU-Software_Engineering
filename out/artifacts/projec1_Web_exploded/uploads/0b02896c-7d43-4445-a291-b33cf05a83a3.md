**GroupBuy 城市社区团购系统**

GroupBuy --- Urban Community Group-buying System

  -------------- ---------------------------
  **选题名称**   GroupBuy 城市社区团购系统
  **难度等级**   ★★★★
  **推荐人数**   4-5 人
  **版本**       v1.0 · 2026-05-06

**一、项目背景**

社区团购在中国一二线城市规模仍在持续增长。以\"团长 + 自提点 +
次日达\"为核心商业模式，本题要求实现一个\"小而美\"的社区团购系统，覆盖商品
/ 团长 / 订单 / 履约 / 结算 五条主线。

**二、业务目标**

> **1.** 消费者通过团长二维码进入社区小店，浏览次日商品；
>
> **2.** 团长在小程序后台管理团员、商品上下架、查看佣金；
>
> **3.** 平台采购方上传商品池、设置佣金规则、调度履约；
>
> **4.** 订单按\"团长 - 社区 - 自提点\"聚合配送；
>
> **5.** 支持次日达 + 自提两种履约方式。

**三、用户角色**

  **角色**   **主要诉求**                   **权限范围**
  ---------- ------------------------------ --------------------------------
  消费者     价格优惠、品质保证、就近自提   浏览 / 下单 / 自提
  团长       佣金可观、操作便捷、客户忠诚   团员管理 / 商品营销 / 提货确认
  平台运营   商品丰富、履约稳定、合规       商品池 / 佣金规则 / 履约调度
  物流       路径优化、时效达标             订单聚合 / 路径 / 签收

**四、功能模块**

**4.1 必做功能（核心交付，≥ 5 个模块）**

> **•** M1 商品中心：商品池 / 标签 / SKU / 库存；
>
> **•** M2 团长中心：注册 / 审核 / 二维码 / 团员管理；
>
> **•** M3 团购小店：消费者端浏览 / 下单 / 支付（沙箱）；
>
> **•** M4 订单聚合：按团长 + 自提点聚合，生成提货单；
>
> **•** M5 履约调度：路径优化（基础贪心）+ 司机端确认；
>
> **•** M6 结算与对账：团长佣金计算 / 平台账单。

**4.2 选做功能（加分项，每实现 1 个 +2 分）**

> **•** LBS 推荐附近团长；
>
> **•** 基于历史订单的销量预测；
>
> **•** 促销引擎：阶梯价 / 拼团 / 限时秒杀；
>
> **•** AI 客服：基于知识库的简单 FAQ。

**五、技术约束**

  **类别**   **约束 / 推荐**
  ---------- -----------------------------------------
  前端       Vue 3 + 微信小程序（团长端 / 消费者端）
  后端       Spring Cloud / NestJS 微服务
  数据库     PostgreSQL 16 + Redis 7
  消息队列   RocketMQ / RabbitMQ（订单聚合）
  地图       高德 / 百度（开发者免费额度）
  鉴权       JWT + 短信验证码（沙箱）

**六、验收标准**

> **1.** 完成 6 个必做模块；
>
> **2.** 能模拟一个\"周二开团 → 周三自提\"的完整闭环；
>
> **3.** 订单聚合算法须输出可解释的聚合理由；
>
> **4.** 团长佣金对账须能与订单流水对得上；
>
> **5.** 提交完整的 12 份产出文件 + 5 份管理表单。

**七、典型风险与建议**

  **风险**         **影响**   **建议**
  ---------------- ---------- -----------------------------------------
  多端复杂度高     高         建议优先完成消费者端 + 团长后台 Web
  路径优化算法     中         采用基础贪心 + 距离优先即可，不强求最优
  佣金规则边界     中         提前梳理佣金规则表 + 单元测试覆盖
  沙箱支付兼容性   低         使用微信支付沙箱 + Mock
